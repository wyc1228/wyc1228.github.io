Git基本使用

1、版本控制

版本控制工具应该具备的功能

  • 协同修改
    • 多人并行不悖的修改服务器端的同一个文件
  • 数据备份
    • 不仅保存目录和文件的当前状态，还能保存每一个提交过的历史状态
  • 版本管理
    • 在保存每一个版本的文件信息的时候要做到不保存重复数据，以节约存储空间，提高运行效率，Git采用了文件系统快照的方式
  • 权限控制
    • 对团队中参与开发的人员进行权限控制
    • 对团队外开发者贡献的代码进行审核–Git独有
  • 历史记录
    • 查看修改人，修改时间，修改内容，日志信息
    • 将本地文件恢复到某一个历史状态
  • 分支管理
    • 允许开发团队在工作过程中多条生产线同时推进任务，进一步提高效率

版本控制工具分类

  • 集中式版本控制工具：CVS、SVN、VSS·····
  • 分布式版本控制工具：Git、Mercurial、Bazaar·····

分布式版本控制工具 比 集中式版本控制工具 首要的优势是 能够避免单点故障

2、Git的优势

  • 大部分操作在本地完成，不需要联网
  • 完整性保证
  • 尽可能添加数据而不是删除或修改数据
  • 分支操作非常快捷流畅
  • 与Linux命令全面兼容

3、Git结构

Git分为工作区，暂存区、本地库

工作区用来写代码，暂存区用来临时存储、本地库存储历史版本

在工作区写完代码通过git add命令临时存储到暂存区，然后git commit提交到本地库

局域网环境下代码托管中心有：GitLab服务器

外网环境下代码托管中心有：码云，github

6、Git命令行操作

6.1 本地库初始化

新建一个文件夹，在此文件夹中打开git bash，输入初始化命令git init

注意：.git隐藏目录下存放的是本地库相关的子目录和文件，不要删除，也不要修改

6.2 设置签名

形式

  • 用户名：wyc
  • Email地址：wang@163.com

作用：区分不同开发人员的身份

辨析：登录远程库(代码托管中心)的账号，密码和这个设置的签名没有任何关系

命令：

  • 项目级别/仓库级别：仅在当前本地库范围内有效
    • git config user.name wyc_pro
    • git config user.email wang_pro@163.com
    • 信息保存位置：.git/config
  • 系统用户级别：登陆当前操作系统的用户范围
    • git config –global user.name wyc_glb
    • git config –globle user.email wang_glb@163.com
    • 信息保存位置：家目录下的gitconfig隐藏文件，~/.gitconfig文件
  • 级别优先级：就近原则，项目级别优先于系统用户级别，二者都有，采用项目级别
  • 二者都没有，不允许这种操作，两种级别最少要设置一个
  • 实际开发中，设置系统用户级别就够了

6.3 基本操作

6.3.1 状态查看

  • git status：查看工作区，暂存区状态
    • On branch master：在master分区下
    • No commits yet：本地库没有东西
    • nothing to commit：暂存区也没有东西

  • 在创建了一个文件之后再git status，还未放入暂存区

    • Untracked files：表示未被追踪的文件

  • 将创建的文件放入暂存区

    • 放入暂存区命令：git add 文件名

  • 将加入暂存区的文件撤回来，只是将暂存区的文件删除，并不会动工作区文件

    • 命令：git rm –cached 文件名

  • 将加入暂存区的文件提交到本地库

    • 提交命令：git commit 文件名
    • 快捷方式：git commit -m ”备注的信息“ 文件名

  • 修改文件之后，再status

6.3.2 查看历史记录

  • 命令：git log

  • 以一行的方式显示：git log –pretty=oneline

  • 显示一部分hash值的命令：git log –oneline

  • 显示回到历史版本需要移动几步的命令：git reflog

    • HEAD@{1}：表示移动到此版本需要移动一步

6.3.3 前进后退(修改历史版本)

  • 基于索引值操作【推荐】，reset既可以前进又可以后退，加哪个索引值就跳到哪个版本

    • git reset –hard 索引值

  • 使用^符号：只能后退，一个^表示后退一步，n个后退n步

    • 往后退一步命令：git reset –hard HEAD^
    • 往后退两步命令：git reset –hard HEAD^^

  • 使用~符号：只能后退

    • 往后退n步命令：git reset –hard HEAD~n

  • reset命令的三个参数对比

    • git reset –hard：在本地库移动HEAD指针，重置暂存区，重置工作区
    • git reset –soft：仅仅在本地库移动HEAD指针
    • git reset –mixed：在本地库移动HEAD指针，重置暂存区

6.3.4 删除文件

  • 将aaa.txt添加到本地库中再删除文件

  • 再查询历史记录，发现指向delete aaa.txt这条记录，回退到上一条new aaa.txt则能重新找回aaa.txt

  • 当删除文件之后提交到暂存区，未提交到本地库中，该如何恢复

    • 命令 git reset –hard HEAD

  • 删除的文件并找回

    • 前提：删除前，这个文件存在时的状态提交到了本地库
    • 操作：用git reset –hard [指针位置]，
      • 删除操作已经提交到本地库：指针位置找历史存在的记录
      • 删除操作尚未提交到本地库：指针位置使用HEAD

6.3.5 比较文件差异

  • 修改apple.txt文件之后比较与原apple.txt之间的差异(添加到暂存区之后就没有区别)

    • 命令：git diff 文件名
      • 将工作区中的文件和暂存区的文件进行比较
    • git diff [本地库中历史版本] 文件名
      • 将工作区中的文件和本地库历史记录比较
    • 不带文件名比较多个文件

7、分支管理

什么是分支？

  • 在版本控制过程中，使用多条线同时推进多个任务

分支的好处

  • 同时并行推进多个功能的开发，提高开发效率
  • 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响，失败的分支删除重新开始即可

命令

  • 查看所有分支：git branch -v

  • 创建一个hot_fix分支：git branch hot_fix

  • 切换分支：git checkout hot_fix

  • 在hot_fix分支修改文件，在提交到暂存区和本地库，此时查看分支状态，发现hot_fix版本往前推进一步

  • 合并分支：

    • 第一步：切换到接受修改的分支上(被合并的)
    • 第二步：执行merge命令
    • 例如，hot_fix分支的修改合并到master上，先切换到master

  • 解决冲突：当两个分支同时修改同一个文件的同一处，而内容又不一致，则当合并后应该遵循哪个修改的，这就会导致冲突

    • 两个分支都修改good.txt文件第一行，全都提交到本地库，则会发现两个分支的所在版本不同，此时合并，修改内容相同则冲突

    • 在hot_fix分支下合并master分支，显示合并失败，此时后面分支为hot_fix|MERGING，需要手动合并

    • 打开修改过后的文件，good.txt

    • 删除特殊符号，修改文件至满意为止

    • 然后git add将good.txt标记为已解决，使用get commit结束合并
    • 注意：此时git commit命令后面不能跟文件名

8、GitHub操作

8.1 创建远程库

  • 登录Github，创建远程库

8.2 在本地创建远程库别名/推送/克隆/拉取

  • 远程库地址

  • 在本地库给远程库地址起别名，以后用别名代替地址比较方便
    • 命令：git remote add 别名 地址

  • 移除远程库别名

    • 命令：git remote remove 别名

  • 推送：git push 别名 分支名，需要输入github账户和密码

    • 此时，已经将本地库中的文件推送到远程库中了

  • 克隆：git clone 远程地址
    • 完整的把远程库下载到本地
    • 创建origin远程地址别名
    • 初始化本地库

注意：现在test00本地库是上传到HaloTest00账号的远程库，而在test01本地库中克隆了test00远程库，此时在test01本地库中修改文件，然后再上传到远程库，因为上传到远程库需要输入账号密码，而继续上传到HaloTest00账号的远程库没有问题，但是如果使用别的账号上传则会报错，因为需要添加团队成员

  • 此时，test01修改文件，提交到本地库没有问题

  • 此时使用HaloTest01的账号密码将本地库推送到HaloTest00的远程库，发现失败，因为HaloTest01未加如HaloTest00团队

  • 将HaloTest01加入到HaloTest00团队，在邮件中同意邀请就可以了

  • 此时HaloTest01加入团队，使用HaloTest01账号推送到远程库

  • HaloTest01修改文件提交到远程库之后HaloTest00拉取远程库
  • 使用git fetch origin master抓取之后，发现工作区的文件内容还是未变

  • 通过git checkout origin/master 命令可以查看下载下来的远程库中文件内容

  • 整个抓取pull操作，分为fetch和merge两步，fetch先下载到本地库(此时未修改工作区文件)看看有没有什么错误，然后merge再合并过来
    • git fetch [远程地址别名] [远程分支名]
    • git merge [远程地址别名/远程分支名]
  • 也可以直接使用pull来操作
    • git pull origin master

8.3 解决冲突

要点：

  • 如果不是基于GitHub远程库的最新版所做的修改，不能推送，必须拉取
  • 拉取下来后如果进入冲突状态，则按照“分支冲突解决”操作解决即可

操作：

  • 首先HostTest00修改文件上传到远程库

  • 此时HaloTest01中用的还是老版本

  • 然后HaloTest01在此基础上修改文件

  • 然后提交到本地库中，准备上传到远程库

  • 上传到远程库，使用HaloTest01账号，因为是团队成员，结果发现上传失败，因为修改冲突

  • 上传不上去，此时提醒你必须先拿到远程库中的最新文件，此时发现进入解决冲突模式

  • 然后将文件修改至满意为止

  • 之后上传到本地库，解决冲突

  • 之后可以再次上传到远程库

10、SSH密匙方式

首先生成密匙，输入此命令

ssh-keygen -t rsa -C “github邮箱”

此时，密匙已经生成到c/Users/王迎春/.ssh/id_rsa.pub文件中

此时复制id_rsa.pub文件的内容，在github中添加key

检查测试连接，执行命令，显示success则成功

ssh -T git@github.com

之后就可以直接使用ssh密匙来clone和push项目，不用输入密码
